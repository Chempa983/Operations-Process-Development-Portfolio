PESISTTEMA

Public Emergency Safety Information System Through Technology

What I built (concept)

I designed PESISTTEMA as a unified municipal safety platform that connects citizens, municipalities, and emergency stakeholders through structured reporting, verified workflows, and an operational dashboard for prioritization and follow-up.

Core modules

How the platform is structured

1) Citizen app

  • Incident reporting with geo-location + photo/video evidence
  • Category selection (traffic, lighting, crime risk, vandalism, hazards, noise, etc.)
  • Severity level guidance (non-urgent vs urgent vs emergency)
  • Status tracking (received ‚Üí assigned ‚Üí in progress ‚Üí resolved)
  • Local alerts and community updates (location-based)

2) Municipal operations dashboard

  • Standardized intake queue with prioritization rules
  • Assignment workflows (ownership + SLA targets)
  • Evidence review & validation flags
  • Internal notes + citizen-facing updates
  • Case closure logic + reporting for audits

3) Emergency logic layer

  • SOS button with automatic location sharing
  • Escalation triggers based on severity and keywords
  • Basic triage guidance (what to do now)
  • Integration-ready structure for emergency stakeholders

4) Analytics & transparency

  • Heatmaps and trends (time, location, category)
  • Response time analysis (SLA compliance)
  • Recurring risk patterns and ‚Äúhot spot‚Äù detection
  • Public transparency summaries (aggregated / anonymized)

Operational workflow

From report to resolution (traceable)

  1. Citizen submits report (category + severity + evidence + location)
  2. System validates (duplicate detection, incomplete fields, basic rules)
  3. Queue prioritization (severity, risk type, proximity to sensitive zones)
  4. Assignment (department / partner owner + SLA timer)
  5. Execution & follow-up (updates pushed to citizen, internal notes logged)
  6. Closure (resolution code, root-cause tag, prevention suggestion)
  7. Analytics (trend and KPI reporting for governance)

Value & KPIs

How impact can be measured

Operational KPIs

  • Average time-to-triage
  • Average time-to-assign ownership
  • SLA compliance by category
  • Re-open rate (quality indicator)
  • Duplicate rate (signal-to-noise control)

Community KPIs

  • Citizen satisfaction after closure (simple rating)
  • Transparency rate (cases with regular status updates)
  • Reduction in recurring incidents by hotspot
  • Increase in ‚Äúearly reporting‚Äù (preventive effect)

Architecture (high-level)

How I would structure the system for scale

  • Front-end: Citizen app + web portal
  • Back-end services: Authentication, case management, notification service
  • Data model: Case entity + evidence + geo + status history + SLA clock
  • Integrations: Municipality ticketing, emergency partner endpoints (future)
  • Security: Role-based access, evidence integrity, anonymized public outputs
